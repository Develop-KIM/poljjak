# 요구사항: 포트폴리오 분석 v2

## 개요 (3문장)

취업 준비 중인 개발자·디자이너가 포트폴리오 PDF를 업로드하면, AI가 직군·연차 맞춤 분석을 수행하고 섹션별 구조화된 Before/After와 우선순위 액션 플랜을 제공한다. 현재 분석 결과가 너무 일반적이고 10개 항목 점수 그리드가 "어디부터 고쳐야 할지" 알 수 없어 실사용 가치가 낮다는 피드백에서 출발한다. 분석 결과를 공개할 때 개인정보(이름·연락처·사진 등)가 노출되지 않도록 AI 자동 익명화도 포함한다.

---

## 사용자 스토리

- As a **신입 프론트엔드 개발자**, I want to **내 포트폴리오의 어떤 프로젝트 설명이 부족한지 원문 맥락과 함께 보고**, So that **막막하게 "고쳐야지" 가 아니라 실제로 고칠 수 있다**.
- As a **주니어 백엔드 개발자**, I want to **원티드 채용공고 URL을 붙여넣고 내 포트폴리오와 비교해서 부족한 부분을 보고**, So that **지원 전에 타겟팅해서 수정할 수 있다**.
- As a **분석 결과 공유하는 사용자**, I want to **공개 시 내 이름·연락처가 자동으로 가려지도록**, So that **개인정보 걱정 없이 커뮤니티에 공유할 수 있다**.

---

## 기능 요구사항

### FR-1: 직군·연차 선택 고도화

- 기존 개발자/디자이너 2가지 → 직군 5가지(프론트엔드/백엔드/풀스택/DevOps/ML엔지니어) × 연차 3단계(신입/주니어/시니어) 조합 선택
- 선택한 직군·연차 조합에 맞는 별도 프롬프트로 분석 (신입 기준 ≠ 시니어 기준)
- 디자이너는 별도 직군 트리로 분리 (UX/UI/브랜드 등)

### FR-2: 분할 화면 Before/After 뷰어

- **레이아웃**: 화면을 좌/우로 분할 (A안 채택)
  - 좌측(Before): PDF.js 또는 iframe으로 원본 PDF 그대로 렌더링
  - 우측(After): AI가 텍스트를 추출·개선한 뒤 유사한 문서 스타일 HTML로 재구성
- **변경 표시 방법** (둘 중 하나 또는 조합):
  - A) 우측(After)에서 변경된 문장·항목에 마우스 호버 시 말풍선으로 "무엇이 왜 바뀌었는지" 표시
  - B) 우측 하단 또는 별도 패널에 "이 부분이 바뀌었어요" 목록 표시 — 클릭 시 해당 위치로 스크롤
- AI가 원본 문서의 섹션 구조(자기소개/프로젝트/기술스택 등)를 자동 감지하여 섹션별로 개선
- 변경되지 않은 부분은 그대로 유지, 개선된 부분만 하이라이트 처리
- 좌/우 동기 스크롤 (한쪽 스크롤 시 반대쪽도 같은 위치로 이동)

### FR-3: 항목별 점수 그리드 폐기 → 우선순위 기반 전체 이슈 표시

- 현재 10개 항목 7~9점 그리드 제거 (사용자가 "어디 고쳐야 해?"를 알 수 없는 구조)
- 개수 제한 없이 AI가 발견한 문제점을 **모두** 나열하되, 우선순위(임팩트 큰 것) 순으로 정렬
- 총점은 유지하되 레이더 차트 또는 간단한 시각화로 변경 (선택 사항)
- 각 이슈에 임팩트 레벨 표시 (예: 높음/중간/낮음) — 개수 제한 대신 레벨로 중요도 전달

### FR-4: 우선순위 액션 플랜 체크리스트

- 분석 완료 후 "이 순서대로 고치세요" 체크리스트 제공
- 각 액션에 예상 임팩트 표시 ("이거 고치면 차별화 포인트 대폭 개선")
- 체크 완료 시 체크 표시 (localStorage 또는 DB 저장)
- 재분석 후 이전 체크리스트와 비교하여 진척도 확인 가능

### FR-5: 분석 비교 기능

- **이전 분석 vs 현재 분석**: 항목별 점수 변화, 새로 추가된/해결된 개선포인트 비교
- **JD 비교**: 원티드·사람인 URL 입력 → 해당 공고 텍스트 크롤링 → "이 포지션 기준 부족한 부분" 하이라이트
- JD 크롤링 실패 시 텍스트 직접 붙여넣기 fallback

### FR-6: 공개 시 개인정보 익명화

- 분석 결과 공개 토글 시 AI가 자동으로 개인정보 감지·마스킹
- 대상: 이름, 이메일, 전화번호, 주소, 사진(얼굴 감지), 깃허브·링크드인 실명 URL 등
- 마스킹 전 미리보기 제공 + 사용자가 수동으로 추가 항목 지정 가능
- 공개된 결과에서는 마스킹 처리된 버전만 표시

### FR-7: 분석 정확도 개선 (프롬프트 재설계)

- 문제 ①: 뻔한 코멘트 반복 → "구체적인 수치를 추가하세요" 대신 원문에서 수치화 가능한 구체 문장 예시 제시
- 문제 ②: 점수 불일치 → 직군·연차별 실제 채용 기준에 맞는 루브릭 설계
- 문제 ③: 직군 무관 분석 → FR-1 직군·연차 조합으로 해결
- 문제 ④: 원문 동떨어진 Before/After → FR-2 섹션 기반 인라인 Before/After로 해결

## 비기능 요구사항

- **NFR-1 (성능)**: 분석 응답 시간 60초 이내 유지 (현재 CLOVA 기준). 섹션 감지·익명화·액션 플랜이 추가되므로 프롬프트 최적화 필요
- **NFR-2 (보안)**: JD 크롤링 시 SSRF 방지 (허용 도메인 화이트리스트: wanted.co.kr, saramin.co.kr 등)
- **NFR-3 (비용)**: 비교 분석(JD/이전 버전)은 토큰 소비가 크므로, 요약 비교 방식으로 최적화
- **NFR-4 (정확도)**: 익명화 누락 시 법적 책임 소지 → 마스킹 후 사용자 확인 단계 필수

## 비목표 (NOT to build)

- After 버전을 실제 PDF 파일로 생성·다운로드 (이번 범위는 HTML 렌더링 뷰어만)
- PDF.js 등 완전한 PDF 렌더러 사용 (원본은 텍스트 추출 후 재렌더링)
- 실시간 스트리밍 분석 (현재 Nitro/CLOVA 구조에서 복잡)
- 포트폴리오 자동 수정·재생성 (AI가 직접 PDF 편집)
- 다른 사용자 포트폴리오와 익명 비교 랭킹
- 원티드·사람인 외 해외 JD 사이트 지원 (LinkedIn, Glassdoor 등)
- 디자이너 직군 세부 분석 (이번 범위는 개발자 직군에 집중)
- 이메일 알림 ("분석 완료됐어요") — 기존 인앱 알림으로 충분

## 가정 (Assumptions)

- CLOVA HCX-005 컨텍스트 길이 내에서 PDF 전문 + 섹션 감지 + 분석이 가능하다
- 원티드·사람인 HTML 구조가 크게 바뀌지 않는다 (크롤러 유지보수 필요)
- 얼굴 감지(사진 익명화)는 간단한 휴리스틱(이미지 태그 위치 등)으로 처리 또는 이 릴리스에서 텍스트 정보만 마스킹

## 미해결 질문 (Open Questions)

- 직군·연차 조합이 15가지(5×3)면 프롬프트도 15개 관리해야 하나? 공통 베이스 + 직군 모듈 방식 고려 필요
- 이전 분석 비교 시 "몇 번까지 이전 분석을 보관"할 건지 (무제한 vs 최근 5개)
- JD 크롤링 실패율이 높으면 텍스트 붙여넣기가 기본이 되어야 할 수 있음
- 액션 플랜 체크 상태를 DB에 저장할지 localStorage로만 처리할지
